# One live roster (PanelGroupMember), PanelMember slimmed to attendance only

Inclusion Panel used to have two overlapping "who's on this panel" tables: `PanelGroupMember` (the Panel Group's standing roster, edited via the shared group modal) and `PanelMember` (a per-meeting copy, shown on Panel Agenda Setup and the live Panel Agenda page). They were never kept in sync — the only reachable "Add Member" UI in the app added to `PanelMember` on the *live* Panel Agenda page, so a newly-created Panel Meeting always started with zero `PanelMember` rows and Setup's Members section showed nothing no matter what the group's actual roster was. `PanelMember` also carried an `attended` boolean that no write path ever set to `False`, duplicating what `checked_in_at`/`left_at` already expressed. `PanelGroupMember`'s `is_active`/`deactivated_at` pair (added the day before this change) had already converged on the same shape `PanelMember` used for roster membership, independent evidence the two models were drifting toward each other rather than serving genuinely different purposes.

`PanelGroupMember` is now the single roster: `_panel_member_roster(panel)` reads it directly for any panel that isn't `complete`, so Setup's Members section and the live Agenda's Panel Members Register both always match whatever the group modal shows, with no sync step to forget. "Add Member" now always means adding to the group (visible on every panel that uses it, present and future) — the live Agenda page's separate per-meeting add-member dialog is gone, replaced by opening the same shared group modal Setup already used. `PanelMember` keeps its name but drops everything except `panel`, a FK to the specific `PanelGroupMember` who checked in, `checked_in_at`, and `left_at` — a row only exists once someone has actually attended. For a `complete` panel, `_panel_member_roster` deliberately reads `PanelMember` instead of the live group roster: `PanelGroupMember` has no history/versioning, so a finished meeting's "who attended" must stay a frozen fact, not something that silently changes if the group's membership changes weeks later. The chair-vacate invariant (deactivating a member who's chairing a panel clears that chair) moved from a dead, unreachable `PanelMember.toggle_active()` method into `toggle_group_member_active` itself, and now explicitly excludes `complete` panels for the same historical-accuracy reason.

## Considered options

- **New `PanelAttendance` model, drop `PanelMember` entirely**: rejected — functionally identical to what shipped, purely a naming choice; keeping the `PanelMember` name avoided a `RenameModel`/`DeleteModel` migration (this app's history had no precedent for either) in favour of a same-app two-step `AddField`+backfill / `RemoveField` pair, matching the existing convention used for `core.Referral`'s backfill (`0004`/`0005`) and `agenda_order`'s backfill (`0009`/`0010`).
- **Snapshot the group's roster into `PanelMember` at meeting-creation time** (restoring something closer to the old `seed_panel_meetings._seed_members` copy, just triggered by the real "New Panel Meeting" flow instead of only the seed command): rejected — reintroduces the exact staleness bug this change fixes, just with a later cutoff; a group roster edited *after* a still-upcoming meeting was created would silently stop being reflected on that meeting's pages again.
- **Keep `PanelGroupMember` and `PanelMember` fully separate, just wire a sync signal between them**: rejected — still two sources of truth to keep consistent (add/remove/expertise-change/deactivate would all need mirroring both ways), for no benefit over reading one table directly; the only place the two concepts *should* diverge is completed-panel history, which a live sync can't express anyway.
