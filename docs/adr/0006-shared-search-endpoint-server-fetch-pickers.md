# Server-fetch, one shared search endpoint, for every Search box over DB data

Search across the portal had drifted into two different architectures doing the same job: Panel's general search (`inclusion_panel_search`) queries the database live per keystroke; the Add Referral student picker and Add Member staff/external picker instead pre-rendered every row for the current scope straight into the page's HTML on load and filtered those DOM nodes client-side in JS. The client-side approach ships the full roster to the browser whether the user ever searches or not, and doesn't scale past a small roster — exactly the one-off-vs-scalable tradeoff this project's scalability priority calls out. It also meant three separate, silently-diverging behaviours for what users experience as "the same search box": Panel search grouped results and hid until typed; the pickers showed everything immediately and never queried the server.

Both pickers now call the same `inclusion_panel_search` view Panel search already used, parameterized by `kind` (`student` / `staff` / `external` / the original `all`), plus `mode`/`school_id`/`exclude` for the staff/external picker's source-toggle and "already on this roster" flagging. Debounce (250ms) and minimum query length (2 characters) are the same across all three callers, and query matching is token-based (each whitespace-separated word must match somewhere in the name) rather than a single substring check, so a two-word query like "Ben Suri" matches split across first/last name. See `PRINCIPLES-INTERACTION.md` (`INT-U13`) for the full behavioural precedent this endpoint now backs.

## Considered options

- **Separate endpoint per picker**: rejected — `kind`/`mode`/`exclude` params on one view cost less than three views independently re-implementing debounce, the 2-char minimum, and token matching, and a future fourth picker (e.g. searching `ExternalContact` from somewhere else) is a new `kind` branch, not a new file.
- **Keep the pickers client-side, just add hidden-until-typed to the existing DOM-filter approach**: rejected — this would still ship the full roster into every page load and leave two genuinely different data-fetching strategies for what's supposed to read as one component; the whole point of standardising was to stop that drift, not just paper over its most visible symptom (results showing before typing).
