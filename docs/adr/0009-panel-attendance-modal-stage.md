# Pre-start attendance is a modal stage on Panel Agenda, not a new screen

#60 shipped a minimal hard block on `start_meeting`: at least one active `PanelGroupMember` must have `checked_in_at` set, enforced server-side, with the toolbar's Start Meeting button disabled + hinted client-side until then. A follow-up grill asked whether attendance deserved a fuller "stage" of its own — a dedicated pre-meeting screen, a full roll call of every member, a low-turnout warning. Every one of those expansions was rejected in favour of what #60 already does (one checked-in member is enough, any member type counts, no soft nudge) — the "at least one" gate itself doesn't change. What does change is *presentation*: attendance moves from a corner of the live Agenda page into a modal the chair must actively engage with before starting.

The existing `schedule-warning-dialog` (an early-start override: "not due yet, reschedule to now?") and the new attendance UI are the same kind of thing — a pre-start gate — so they merge into one modal. It auto-opens on every load of Panel Agenda while `panel.started_at` is `None`, regardless of whether the panel is early, on time, or delayed. If the panel isn't due yet, a reschedule prompt appears at the top of the modal; the full member roster (photo, expertise, check-in toggle — the same full-row treatment the page already used) and the Start Meeting button (disabled + hinted until the block is satisfied) always appear below it. "View Agenda" dismisses the modal so the chair can preview Pending/Discussed underneath without taking attendance; the toolbar's Start Meeting button and a placeholder in the Members Register column ("N of M checked in — Take Attendance") both reopen it. Outside the modal, pre-start, the Members Register column is read-only — a count and a reopen trigger, not a second set of live toggles — so there is exactly one place attendance gets set before the meeting starts.

Once running, attendance stays exactly as editable as #60 already made it (late arrivals, leavers, no locking) — only the layout of the Members Register column changes, splitting into **In Attendance** (checked in, not left) and **Not In Attendance** (never checked in, or checked in then left) groups, each row keeping its toggle.

No model or migration changes: this is presentation only, over the `checked_in_at`/`left_at` fields `PanelMember` already had after #60/[0005](0005-merge-panel-membership-into-panelgroupmember.md).

## Considered options

- **A dedicated `/meetings/<id>/attendance/` screen the chair is routed through before Agenda**: rejected — it's a whole new route for what is functionally "the top of the Agenda page before Start is clicked." A separate screen also reopens the staleness problem #60's design deliberately avoided in `inclusion_panel_meeting_start` (see #60): the chair would need to navigate away from and back to the same panel state, rather than the modal simply reflecting whatever the page's own render already computed.
- **Keep the live check-in/mark-left toggles rendered on the page pre-start, in addition to the modal**: rejected — two controls doing the exact same thing (one in the modal, one in the sidebar) invites divergent-looking state and forces the chair to reason about which one is "real." Locking the page's own pre-start toggles and making the modal the sole place attendance is set removes that ambiguity for free.
- **A three-way running-state split (In Attendance / Left / Never Checked In)**: rejected in favour of two groups (In Attendance / Not In Attendance) — the extra distinction between "attended then left" and "never showed" is real but not worth a third layout section; both are equally "not currently in the room," which is what the split is for.
