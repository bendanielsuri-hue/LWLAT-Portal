# Portal Interaction Language

Motion, hover/focus/selected states, and transition rules — split out from [DesignLanguage.md](DesignLanguage.md), which covers static visual rules (colour, layout, typography). Extracted from the Inclusion Panel — the most complete, stable UI in the codebase. Apply these rules to all future hub pages.

---

## Interaction patterns

**Hover on a selectable row**
`background: var(--row-hover-bg)` (= `color-mix(in srgb, var(--text-primary) 6%, var(--bg-surface))`) + title weight bumps to 700. Apply via `.selectable:hover` or explicit `.panel-list li:hover` / `tbody tr:hover`.

**Chosen / selected state on a row** (persistent after click)
`background: var(--primary-light); box-shadow: inset 3px 0 0 var(--accent-border)` + title weight 700. Hover on a chosen row keeps `--primary-light` — does not revert to `--row-hover-bg`. Applied via `.chosen` on `entity-row`, `panel-list li`, `agenda-table tbody tr`.

**Hover and chosen on a selectable card** (`.meeting-card`, and any future card-shaped — as opposed to row-shaped — selectable)
Cards stay off `--primary`/`--accent-border` entirely; the accent colour reads as too strong at card scale, where the whole tile (not a thin row) carries the fill. Both states share one background token so a selected card already reads as "hover-lit":
- Hover: `border-color: var(--border-strong); box-shadow: var(--shadow-sm), inset 0 0 0 1px var(--border-strong); background: var(--meeting-card-hover-bg)` + title weight 700.
- Chosen: same `background: var(--meeting-card-hover-bg)`, `box-shadow: var(--shadow-sm), inset 3px 0 0 var(--border-strong)` + title weight 700.
- Chosen *and* hovered: layer both insets — `box-shadow: var(--shadow-sm), inset 3px 0 0 var(--border-strong), inset 0 0 0 1px var(--border-strong)` — so a selected card still gives tactile feedback on hover instead of looking inert. Needs an explicit `.chosen:hover` rule since `.chosen` and `:hover` are equal-specificity classes and `.chosen` alone would otherwise always win by source order.
- `--border-strong` (not a fixed neutral like `--text-secondary`) is the right token here because it's already redefined per theme flavour in `themes.css` — the emphasis stays "on-theme" without touching `--primary`.

**Search** (behavioural precedent — visual chrome for `.app-search`/`.app-search-field` is in DesignLanguage.md's own "Search" entry)
Every Search box in the app — the Global Rail's app search, the homepage hero search, Panel's general search modal, Add Referral's student picker, Add Member's staff/external picker — follows the same behaviour, not just the same border/icon styling. This precedent grew out of Panel search (the most-liked implementation) and standardised the others to match it:
- **Hidden until typed, no exceptions.** A Search box shows nothing — no result list, no "browse everything" default — until the user types. This applies even to pickers that used to show their full option list up front (Add Referral, Add Member): there is no browse-without-typing mode anywhere in the app. Reserve the space with a placeholder `.empty-note` ("Start typing to search…") rather than a collapsed empty container — see the "growable modal" empty-state rule above.
- **Server-fetch for anything backed by real DB data.** Students, staff, external contacts, referrals, and any future DB-backed entity are queried live per keystroke against the shared `inclusion_panel_search` endpoint (`hubs/inclusion/panel/views.py`, one view parameterized by `kind`), not pre-rendered into the page and filtered client-side — see `docs/adr/0006-shared-search-endpoint-server-fetch-pickers.md` for why. Only genuinely static, small client-side data (the Global Rail's list of hub/page links, embedded once via `json_script`) stays client-side; it isn't a query against growing data and never will be.
- **250ms debounce, 2-character minimum**, same numbers everywhere. Below 2 characters, show a "Keep typing… (2+ characters)" hint rather than either querying or staying silent.
- **Token-based matching.** The query is split on whitespace; every token must match somewhere in the name (first name, last name, or — for single-field names like `ExternalContact` — the one name field). "Be S" matches "Ben Suri" because "Be" matches the first name and "S" matches the last, independently. A single-word query keeps working exactly as a plain substring match always has.
- **Group-by-kind headers only when a single search can legitimately return more than one entity type.** Panel's general search spans students and staff, so it groups under "Students"/"Staff" labels. Add Referral (students only) and Add Member (staff/external, but never both at once — the source segmented control already picks one) never show a kind header; there's nothing to disambiguate.
- **No outer card around the result list.** Two different "app search" looks exist in the codebase — the homepage hero search's floating bordered/shadowed card (`.app-search-results`' default rules, `navigation.css`), and the Global Rail's Sidebar Search overlay, which drops the border/radius/shadow entirely and lets rows sit directly in the panel's own flow (`.side-nav .app-search-results`). Pickers follow the **Sidebar overlay** look, not the floating card: no wrapping `border`/`border-radius`/`box-shadow` around the result list, and the container sizes to its actual content via `max-height` (not a fixed `height` that reserves dead space when there are few results) — each row reads as its own rounded "card" only once hovered/active (`border-radius` + `background` on `:hover`/`.active`, no border between rows), not because the whole list sits inside a permanent box. Getting this wrong reads as "a nested card" — a big, mostly-empty bordered rectangle sitting under the already-bordered search input (see `.referral-student-results`/`.member-result-list` in `hubs/inclusion/panel/static/css/panel.css` for the corrected reference).
- The one search surface this does **not** (yet) cover is the Students/Referrals/Actions list pages' own "Name Search" filter-bar field — still a plain `.filter-field` that live-narrows the page's own table, combined with Year/Reg. That field is slated to move to this same dropdown-results pattern too; how a selected result should interact with the page's other filters is still an open question, deliberately deferred rather than guessed at here.
- **Switching a picker's source (the segmented control above Add Member's Search box) keeps whatever's typed and re-queries it against the new source, rather than clearing it.** Only a genuinely fresh pick — the "Change" button on an already-selected member/contact — resets the search text to empty. Clearing on every source switch used to force a retype for what's usually a "wrong tab, not wrong query" correction (`hubs/inclusion/panel/static/js/panel.js`'s `initMemberPicker`, `showPicker(keepSearchText)`).

**Fade toggle** (`window.setFadeHidden`, `hubs/inclusion/panel/static/js/panel.js`; CSS in `panel.css` next to `dialog.modal-dialog`)
For a small, same-size element that pops in/out of an already-open dialog without changing its height — a footer button swapping for another (Add Member ↔ Back ↔ New External Contact) — fade the opacity instead of flipping `hidden` and letting the `display: none` UA rule snap it away instantly. Call `window.setFadeHidden(el, hide)` rather than assigning `el.hidden` directly. This is the small-element counterpart to "Modal content swap (height transition)" above, not a replacement for it — a change that alters the dialog's *height* (a whole section appearing, a list growing) still goes through `window.animateModalHeightChange`, not this. Reach for whichever one matches what's actually changing: pure visibility of one small thing → fade toggle; a height-affecting section swap → the height-transition helper; both at once (rare) → wrap the height-transition mutate callback around calls to this.

**Focus visible**
`outline: 2px solid var(--primary); outline-offset: 2px`. Never suppress focus outlines.

**Row transition — none, deliberately**
Hover/focus/chosen state changes on rows/cards/tabs/nav (background, border-color, box-shadow, color) apply with no `transition` at all — they should read as immediate, not a fade. This applies to `.selectable`, tabs (`.tab-row`, `.card-tab`, `.tab-row-more-btn`), breadcrumbs, the nav rail, `.agenda-table tbody tr`, `.panel-list li`, etc. `.btn` (and its variants) is the one deliberate exception — it keeps a soft `transition: background var(--transition-slide), border-color var(--transition-slide), color var(--transition-slide)` on hover/focus (the slower 360ms token, not `--transition-fast`), since a snappier button hover read as too harsh. Beyond that, smooth `transition`/`animation` is reserved for things that are genuinely animating — an element entering/exiting, moving, or resizing (modal open/close, sidebar collapse, row grow-in/shrink-out, the tab-underline pop-in, the discussing pulse) — not for a plain colour/background swap on `:hover`. The handful of transform-based hover micro-interactions below (chip lift, swatch scale, nav icon nudge) are movement, not a fade, so they keep their transition.

**Modal open/close animation**
`opacity: 0; transform: translateY(8px) scale(0.97)` → `opacity: 1; transform: translateY(0) scale(1)`. Duration `450ms`. Backdrop: `rgba(15,23,36,0.55)`, transitions opacity. Dark theme backdrop: `rgba(0,0,0,0.7)`.

**Modal content swap (height transition)** (e.g. New Referral's student-picker ↔ question-fields swap, Panel Group's inline "Create Panel Group" swap — see DesignLanguage.md's "Inline creation swap")
**General rule: any modal content change that alters the dialog's natural height goes through this, full stop** — not just an explicit view/step swap (list ↔ add-member, search ↔ selected-row), but also a live filter that hides/shows some of an *already-rendered* set of rows (typing into a search box that shows/hides matches) and a live result count that changes how much content there is to show at all (Panel Search's own result list). If a change can make the dialog taller or shorter, wrap it — the size of the change doesn't matter, a single row's worth snapping is just as noticeable as a whole section's worth. The one exception is a swap happening *before* `dialog.showModal()` — `window.animateModalHeightChange` no-ops itself when the dialog isn't open yet (see its own `!dialog.open` guard), so wrapping is harmless but pointless there.

A modal that shows/hides different sections of its own content in place (not opening/closing — the dialog stays open throughout) eases to its new height instead of snapping, via `window.animateModalHeightChange(dialog, mutate)` (`hubs/inclusion/panel/static/js/panel.js`): snapshot the dialog's current rendered height as an explicit pixel value, run `mutate` (the actual DOM change), then on the next frame measure the new natural height (`scrollHeight`) and transition to *that* explicit value — clearing back to `height: auto` once the transition's had time to finish, so a later swap isn't pinned to a stale pixel height. This works because `dialog.modal-dialog` already carries a `height` transition in its base rule (`--modal-duration`, the same `450ms` as open/close) that nothing else uses — `height: auto` itself can't be transitioned, which is why the explicit-pixel snapshot/measure dance is needed at all. Wrap *every* content mutation this way, including a later "loading → loaded" replacement within the same swap, not just the initial show/hide — a second untransitioned jump right after the first undoes the smoothness.

Not every dialog that uses `animateModalHeightChange` should also be top-anchored, though — that's a separate, narrower decision. A dialog whose content is likely to change height **repeatedly** while open (adding/removing rows, a live search-filtered list re-narrowing on every keystroke) is a **"growable" modal** — currently Panel Search, Panel Group Edit, and New Referral (its student search re-filters live as you type, even though the picker-vs-form step it sits inside is itself a one-off switch). Anchor these near the top of the viewport (`top: 8vh; bottom: auto; margin: 0 auto`, see `dialog#panel-search-dialog[open]` and its sibling selectors in `panel.css`) instead of the shared `dialog.modal-dialog`'s vertical centering — growing/shrinking height while vertically centered shifts the *top* edge on every change (the dialog visibly walks up and down the screen), whereas anchoring near the top keeps the top edge fixed and only the bottom edge moves, which reads as one dialog resizing rather than the whole thing relocating.

A dialog that merely switches once between a couple of fixed modes — Panel Meeting/Settings' inline "Create Panel Meeting" ↔ "Create Panel Group" swap is the reference case — still animates that one height change, but stays **centered**. It only ever resizes once or twice per open, so centering's usual "re-centers as it resizes" downside barely shows, and it keeps the same vertical placement as every other simple modal in the app rather than sitting in a visibly different spot for what's ultimately a single form. Judge each dialog by which pattern its content actually follows, not by whether `animateModalHeightChange` appears anywhere in its code — that alone doesn't make it a "growable" modal. This is a deliberate, scoped override per growable dialog, not a change to the shared base rule — small static dialogs (Add Expertise Tag, confirm dialogs) whose content never changes shape stay centered too.

A results/list area that reserves no minimum height for its own empty/idle state (e.g. before a search has a query) will visibly collapse the instant the dialog opens, then grow again once there's something to show — settling twice instead of once. Prefer rendering *some* placeholder content for that state (an `.empty-note` "Start typing to search…", not an empty string) over a bare CSS `min-height` hack — it reserves the same space while actually telling the user something, and needs no separate value to keep in sync with what the populated state actually looks like.

**Row grow-in / shrink-fade-out** (adding/removing an item from a visible list)
Animate `max-height`/`opacity`/`padding-top`/`padding-bottom`/`margin-top`/`margin-bottom` together via `Element.animate()` (not a CSS transition — CSS transitions intermittently snap straight to the end state on freshly-inserted/removed rows instead of animating). Duration `900ms`, easing `cubic-bezier(0.4, 0, 0.2, 1)`, same properties/timing both directions (values simply reversed). Pair with an instant colour-tint flash (`.agenda-flash-added`/`.agenda-flash-removed`, `color-mix(...)` at 12% of `--color-positive`/`--color-negative`) for add/remove feedback. Use for `div`/`li` rows spaced by padding + `border-bottom` (e.g. `.entity-row`, `.settings-row`) — not for `<table>` rows (browsers don't animate table-row height cleanly) and not for margin/`gap`-spaced lists without adapting the technique first (gap won't collapse in lockstep with the animated row). Helpers (`shrinkAndFadeOut`, `growIn`, `flash`, `cancelRowAnim`, `wireRowRemoveForm`) live as top-level functions in `hubs/inclusion/panel/static/js/panel.js`, above `initAgendaDragDrop` — reuse them directly rather than reimplementing per hub.

**Refreshing a live fragment in place** (e.g. re-fetching a modal/panel's HTML after an autosave and swapping it back in)
A full `container.innerHTML = freshHtml` replace is fine for the *first* render of a fragment, but doing it on every subsequent update tears down and recreates every element inside — which silently reintroduces two problems: (1) any scrollable descendant (`overflow-y: auto`) loses its scroll position, since the freshly-created element always starts at `scrollTop: 0`, reading as the panel "jumping to the top"; (2) any element whose entrance animation is a plain unconditional CSS `animation` (not gated by JS), e.g. `.tab-row`'s `tab-row-fade-in`, replays that animation every time it's recreated, reading as a flash/flicker on an update rather than a one-off entrance. When a container is refreshed (not opened for the first time), capture the scroll position of its scrollable descendant(s) before replacing `innerHTML` and restore it after, and suppress replay of one-shot entrance animations on refresh renders (e.g. track an `hasRenderedOnce`-style flag per session and set `el.style.animation = 'none'` on the affected elements when true) — see the Panel Group modal's `render()` in `hubs/inclusion/panel/static/js/panel.js` for the reference implementation. A true DOM diff/patch avoids both problems more thoroughly but is a bigger lift than this codebase's fragment-replace pattern warrants for most cases — reach for it only if a container needs to preserve more than scroll position (e.g. focus, in-progress row animations). A third failure mode in the same family: if the server-rendered fragment conditionally omits a structural element once its underlying data hits zero (e.g. a tab whose panel only renders `{% if inactive_members %}`), then whatever emptied that data (the very action the user just took) yanks that structure out of the refreshed fragment, forcing the UI to fall back to some default view — reading as an involuntary "it switched tabs on me." Render the structure unconditionally (with its own empty state, e.g. "No inactive members") whenever it's paired with a sibling that's still populated, rather than gating its existence on the count that the user's own action just changed.

**Modal header close button** (`.modal-close`)
Every `dialog.modal-dialog` gets one: a plain `&times;` button (`font-size: var(--font-3xl); color: var(--text-faint)`, hover `--text-primary`, no border/background) as the last child of `.modal-header`, which lives outside the scrolling body — so it's always visible, even after scrolling. Wired generically via `data-modal-close` (the same attribute every in-body Cancel/Close button already uses) — for a dialog whose click listener is scoped to itself (`dialog.addEventListener('click', ...)` checking `e.target.closest('[data-modal-close]')`), adding the button is template-only, no JS needed. **A modal's bottom row keeps its own Close/Done button only when there's at least one other action button next to it** (Cancel+Save, Cancel+Create, Yes+No). A solitary dismiss button at the bottom — nothing but "Close" or "Done" — is redundant with the header X and gets removed entirely, not kept as a second way to do the same thing.

**Discussing pulse** (`.status-pill.discussing`)
`animation: discussing-pulse 2s ease-in-out infinite` — oscillates opacity 1 → 0.6. Only on the active-discussion state; do not use for general attention-drawing.

**Status-filter tab entering/leaving the tab row** (a tab whose count crosses to/from zero — see DesignLanguage.md's "Status-filter tab rows")
A tab still fully disappears at count 0 — no disabled/greyed state exists for this — but the disappearance/appearance itself eases instead of snapping: `opacity` fades to 0 together with `max-width`/`padding`/`margin` collapsing to 0, ~260ms `ease`, reversed on the way back in. No translate/scale — this is a collapse, not a pop, so it doesn't compete with the tab-underline's own pop-in.

**Count-delta pulse** (any live `(N)` changing while it's already visible — a status tab's own count, a card/section heading's total, e.g. an action moving from Incomplete to Complete, or a referral being removed from an agenda)
Fires live, tied to the action that caused the change — not on a page load/fragment refresh rediscovering a different count. Not limited to tab labels: `.panel-card-count`/`.setup-col-header` heading counts get the same treatment (Home's My Referrals/My Actions, Panel Agenda Setup's Referral Selection tabs and Panel Agenda heading, Panel Agenda's Students Pending heading). The whole `(N)` — brackets included, not just the digit — scales asymmetrically and tints, then settles back to normal:
- Increase: scale peaks at `1.32`, colour tints `--color-positive`, bounce easing (`cubic-bezier(.34, 1.56, .64, 1)`).
- Decrease: scale dips to `0.78`, colour tints `--color-negative`, plain `ease`.
- Both settle back to `scale(1)` / inherited colour over `750ms` total.
- Applies to every status tab affected by the change, including the `All (N)` aggregate tab — it's still a live count, not exempt just for being the default.
- No background box (reads "boxy" at tab-row text scale) and no arrow-chip/odometer alternative — scale + colour on the number itself was preferred over both after side-by-side comparison.
- Implementation: `window.pulseCount(el, 'up'|'down')` (`panel.js`) adds/removes the `.count-pulse-up`/`.count-pulse-down` classes (`panel.css`) on whichever element holds the number — a tab's `.count` span, or the heading's own `.panel-card-count`/`.count` element. A page with its own drag-and-drop/fragment-swap flow (Panel Agenda Setup, Panel Agenda) doesn't call this directly — it marks each countable element with `data-count-key`/`data-count`, and `initAgendaDragDrop`'s shared `applyFreshDoc` (`panel.js`) diffs old vs. new automatically on every column swap. A page recomputing counts purely from remaining DOM rows (e.g. Home's My Referrals after a delete, no server round-trip needed) uses `window.recountTabsFromRows` instead, which calls `pulseCount`/`setTabCollapsed` itself. A modal that already re-renders itself via its own `render(html)` function (Edit Panel Group's Active/Inactive Members tabs, `panel.js`) snapshots each `.count`'s `data-count` before the old markup is torn out and diffs against the freshly-rendered one after — same idea as `applyFreshDoc`'s diff, just inline in that modal's own render rather than the shared drag-drop helper. Skipped on the render that first opens the modal (nothing to compare against yet) and on a tab that just faded in via its own `.tab-fade-in` entrance (gaining a first member is already its own "something changed" signal — pulsing the count too would double up on the same moment).
- Genuinely can't fire everywhere a count appears — a page whose only mutating actions cause a full navigation (Panel Meetings' Draft/Ready/Running/Delayed/Complete status tabs: Start/Delete both navigate away or reload) has nothing in-page to animate. Don't force a pulse onto a page load's first render just to "cover" it.
- **General rule: a count pulse fires the moment its own action is confirmed — never deferred to wait for a row/column's own removal-or-move animation, or for a re-render that's itself gated on that animation finishing.** Several places in this app hold a DOM swap/re-render off until a row's local shrink-out (`shrinkAndFadeOut`, ~900ms) has finished, precisely so the swap doesn't cut that row's own animation short (`beginColumnAnim` in `panel.js` for Panel Agenda Setup/Panel Agenda's drag-and-drop; the `Promise.all([fetchPromise, shrinkPromise])` gate on Edit Panel Group's Active/Inactive toggle). That gate is correct for the *DOM swap* — but a count pulse riding along on that same swap inherits its delay too, and reads as a laggy, second-class citizen next to an increase elsewhere that swaps immediately (nothing gates the column/tab a row is *arriving* at). Whenever what a count is about to become is fully knowable client-side the instant the triggering click lands (a row leaving a column always means that column's count drops by exactly one; a member toggling Active↔Inactive always means one tab's count rises by one and the other falls by one) — bump and pulse it right there, synchronously, ahead of the fetch/animation/re-render. The eventual gated swap's own before/after diff (`applyFreshDoc`, or a modal's own `render()`) will then see the already-bumped number agree with the freshly-fetched one and stay quiet, rather than pulsing a second time. See `optimisticallyDecrementCounts` (drag-and-drop) and the `toggle_group_member_active` handler (Edit Panel Group) in `panel.js` for the two current instances of this same rule.

**`prefers-reduced-motion` scope**
The first use of this media query in the codebase — everything above predates it. It suppresses motion that's *attached to an interactive/hoverable element* (the chip lift, swatch scale, and text-size-option lift under "Micro-interactions" above; the nav icon nudge) — motion a user triggers just by hovering something clickable. It does **not** suppress the tab count-delta pulse above: that's a passive indicator reporting a change that already happened, not a hover effect on something to click, so it keeps its animation regardless of the setting.

**Micro-interactions**
Priority chips: `transform: translateY(-1px)` on hover. Colour swatches: `transform: scale(1.12)`. Text-size options: `transform: translateY(-2px)`. These are the only elements that use transform-based hover — do not add hover transforms to buttons or cards.

**Input focus ring**
`border-color: var(--primary); box-shadow: 0 0 0 3px color-mix(in srgb, var(--primary) 25%, transparent)`.

## Anti-patterns

1. **Don't invent a new selected-state colour**. Row selectables always use `background: var(--primary-light); box-shadow: inset 3px 0 0 var(--accent-border)`; card selectables always use `background: var(--meeting-card-hover-bg); box-shadow: inset 3px 0 0 var(--border-strong)` (see "Hover and chosen on a selectable card"). A different fill or an ad-hoc neutral (e.g. `--text-secondary`) breaks visual consistency with every other selectable of that shape in the app. Popover/dropdown options (`.ui-option.selected`, `components/forms.css`) are the one deliberate exception to the accent-bar half of this rule: same `--primary-light` fill, but no `--accent-border` bar — a popover option is already corner-clipped to the popover's own rounded corners, and a left bar reads oddly against that clipped edge.

2. **Don't apply `transform: translateY` or `scale` hover effects to buttons or cards**. Lift/scale transforms are reserved for chips, swatches, and text-size options. Adding them to buttons or cards makes the UI feel unstable.

3. **Don't suppress `focus-visible` outlines**. The design relies on `outline: 2px solid var(--primary); outline-offset: 2px` for keyboard accessibility. Never set `outline: none` without an equivalent replacement.

4. **Don't blindly `innerHTML`-replace a live fragment on every update without preserving scroll position and suppressing one-shot entrance animations**. See "Refreshing a live fragment in place" above — an unguarded refresh reads as a jarring "the whole panel just reloaded," even though no navigation happened.

5. **Don't stack a full-weight creation modal on top of an already-open one.** A "+" clicked from inside a modal that opens a *second*, heavier modal (its own multi-field form, its own header/footer) reads as "a small modal on top of a small modal." When the thing being added is itself a small, single-field affair (e.g. the shared "Add Expertise Tag"/"Add External Contact" dialogs), a second stacked `<dialog>` is fine — it's genuinely lightweight. When it's a fuller creation flow with several fields (e.g. Panel Group's "Create Panel Group"), swap the *host* modal's own content in place instead — see DesignLanguage.md's "Inline creation swap" and this file's "Modal content swap (height transition)" above. Judge by weight, not by a blanket "never nest dialogs" rule.

6. **Don't let a live filter or result count snap a modal's height.** A search box that hides/shows rows as you type, or a live result list that replaces its own content, changes the dialog's natural height exactly as much as an explicit view/step swap does — it needs the same `window.animateModalHeightChange` treatment, not just the named "modes" a dialog switches between. See "Modal content swap (height transition)" above — this was previously missed on New Referral's student-filter typing and the Panel Group Add Member picker's search-vs-selected toggle, both fixed by wrapping the same way their neighbouring step-transitions already were.

7. **Don't show a Search box's full result set before the user has typed anything.** Add Referral and Add Member both used to render every student/staff/external option up front and merely filter it client-side as you typed — the opposite of Panel search's hidden-until-typed behaviour. Standardised: every Search box is empty until typed, no picker gets a "browse everything" exception. See "Search" above.

8. **Don't pre-render a full DB-backed list into the page and filter it in JS.** Shipping every student/staff/external row to the browser on load so a search box can filter the DOM client-side doesn't scale past a small roster and duplicates the server-fetch behaviour every other Search box already uses. Query the shared `inclusion_panel_search` endpoint instead — see "Search" above and `docs/adr/0006-shared-search-endpoint-server-fetch-pickers.md`.
